# Google Secret Manager With External Secrets Operator

Last verified: 2026-07-02

This guide explains how to pass application secrets from Google Secret Manager to pods deployed by ArgoCD and the shared Helm chart in this repo.

The target design is:

```text
Google Secret Manager
  stores real secret values

External Secrets Operator
  runs in each application cluster: dev, uat, prod

ClusterSecretStore or SecretStore
  Kubernetes resource that tells ESO how to authenticate to Google Secret Manager

ExternalSecret
  Kubernetes resource generated by this repo's Helm chart per app

Kubernetes Secret
  generated automatically by ESO in the app namespace

Deployment
  reads the generated Kubernetes Secret through envFrom or secretKeyRef
```

Important correction: Google Secret Manager does not contain a `SecretStore`. The `SecretStore` or `ClusterSecretStore` is a Kubernetes custom resource created inside each target cluster.

## 1. What Runs Where

You described four clusters:

```text
argocd-cluster
dev-cluster
uat-cluster
prod-cluster
```

Install External Secrets Operator in the clusters where application pods run:

```text
dev-cluster  -> install ESO
uat-cluster  -> install ESO
prod-cluster -> install ESO
```

You do not need ESO in the ArgoCD cluster unless applications also run there.

ArgoCD can still manage ESO remotely. ArgoCD lives in `argocd-cluster`, but it applies manifests into `dev-cluster`, `uat-cluster`, and `prod-cluster`.

## 2. Recommended Naming Convention

Use predictable names because you will have 30+ apps.

Google Secret Manager secret names:

```text
<app>-<secret-purpose>
```

Examples:

```text
spring-boot-realworld-jwt-secret
spring-boot-realworld-db-password
spring-boot-realworld-redis-password
```

Kubernetes generated Secret name:

```text
<app>-secrets
```

Example:

```text
spring-boot-realworld-secrets
```

Use this env-free naming style when dev, uat, and prod should all read the same secret value from Google Secret Manager. If any secret value must become different per environment later, add the environment back only for that specific secret.

Keys inside the generated Kubernetes Secret should match application env var names:

```text
JWT_SECRET
DB_PASSWORD
REDIS_PASSWORD
```

## 3. Prerequisites

You need:

```text
gcloud CLI access to the Google Cloud project
kubectl access to dev, uat, and prod clusters
helm CLI, if bootstrapping ESO manually
ArgoCD access, if installing ESO through ArgoCD
GKE Workload Identity Federation enabled on target clusters
Secret Manager API enabled in the Google Cloud project
```

Set common variables before running commands:

```bash
export PROJECT_ID="montra-490412"
export PROJECT_NUMBER="$(gcloud projects describe ${PROJECT_ID} --format='value(projectNumber)')"
export REGION="asia-south2"
```

Enable the Secret Manager API once per Google Cloud project:

```bash
gcloud services enable secretmanager.googleapis.com --project="${PROJECT_ID}"
```

## 4. Create Secrets In Google Secret Manager

Each Google Secret Manager secret stores the actual value. The repo stores only the secret name.

Create a secret:

```bash
gcloud secrets create spring-boot-realworld-jwt-secret \
  --project="${PROJECT_ID}" \
  --replication-policy="automatic"
```

Add the first version:

```bash
printf '%s' 'replace-with-real-jwt-secret-value' | \
  gcloud secrets versions add spring-boot-realworld-jwt-secret \
    --project="${PROJECT_ID}" \
    --data-file=-
```

For rotation, add a new version to the same secret:

```bash
printf '%s' 'new-secret-value' | \
  gcloud secrets versions add spring-boot-realworld-jwt-secret \
    --project="${PROJECT_ID}" \
    --data-file=-
```

ESO reads the latest enabled version by default unless you configure a specific version.

## 5. Enable Workload Identity Federation For GKE

For GKE Autopilot, Workload Identity Federation for GKE is enabled by default.

For GKE Standard clusters, enable it at the cluster level:

```bash
gcloud container clusters update DEV_CLUSTER_NAME \
  --location="${REGION}" \
  --workload-pool="${PROJECT_ID}.svc.id.goog" \
  --project="${PROJECT_ID}"
```

If existing node pools were created before Workload Identity was enabled, update each node pool:

```bash
gcloud container node-pools update NODEPOOL_NAME \
  --cluster=DEV_CLUSTER_NAME \
  --location="${REGION}" \
  --workload-metadata=GKE_METADATA \
  --project="${PROJECT_ID}"
```

Repeat for `uat-cluster` and `prod-cluster`.

## 6. Create A Google Service Account Per Environment

Use one Google service account per environment. This keeps dev, uat, and prod access isolated.

Dev:

```bash
gcloud iam service-accounts create external-secrets-dev \
  --project="${PROJECT_ID}" \
  --display-name="External Secrets Operator Dev"
```

UAT:

```bash
gcloud iam service-accounts create external-secrets-uat \
  --project="${PROJECT_ID}" \
  --display-name="External Secrets Operator UAT"
```

Prod:

```bash
gcloud iam service-accounts create external-secrets-prod \
  --project="${PROJECT_ID}" \
  --display-name="External Secrets Operator Prod"
```

## 7. Grant Secret Manager Access

Simplest option:

```bash
gcloud projects add-iam-policy-binding "${PROJECT_ID}" \
  --role="roles/secretmanager.secretAccessor" \
  --member="serviceAccount:external-secrets-dev@${PROJECT_ID}.iam.gserviceaccount.com"
```

Repeat for UAT and prod with their own service accounts.

More secure option:

Grant access only to specific app secrets:

```bash
gcloud secrets add-iam-policy-binding spring-boot-realworld-jwt-secret \
  --project="${PROJECT_ID}" \
  --role="roles/secretmanager.secretAccessor" \
  --member="serviceAccount:external-secrets-dev@${PROJECT_ID}.iam.gserviceaccount.com"
```

For production, prefer per-secret IAM or a separate production Google Cloud project.

## 8. Install External Secrets Operator In Each Target Cluster

First connect `kubectl` to the target cluster:

```bash
gcloud container clusters get-credentials DEV_CLUSTER_NAME \
  --location="${REGION}" \
  --project="${PROJECT_ID}"
```

Install ESO:

```bash
helm repo add external-secrets https://charts.external-secrets.io
helm repo update

helm upgrade --install external-secrets external-secrets/external-secrets \
  --namespace external-secrets \
  --create-namespace \
  --set installCRDs=true \
  --set serviceAccount.annotations."iam\.gke\.io/gcp-service-account"="external-secrets-dev@${PROJECT_ID}.iam.gserviceaccount.com"
```

Repeat this installation in `uat-cluster` and `prod-cluster`, changing the Google service account:

```text
dev-cluster  -> external-secrets-dev@PROJECT_ID.iam.gserviceaccount.com
uat-cluster  -> external-secrets-uat@PROJECT_ID.iam.gserviceaccount.com
prod-cluster -> external-secrets-prod@PROJECT_ID.iam.gserviceaccount.com
```

Verify ESO is running:

```bash
kubectl get pods -n external-secrets
kubectl get crds | grep external-secrets
```

Expected result:

```text
external-secrets pods are Running
externalsecrets.external-secrets.io CRD exists
secretstores.external-secrets.io CRD exists
clustersecretstores.external-secrets.io CRD exists
```

## 9. Allow The ESO Kubernetes Service Account To Impersonate The Google Service Account

ESO's Kubernetes service account is usually:

```text
namespace: external-secrets
name: external-secrets
```

Grant it permission to use the Google service account.

Dev:

```bash
gcloud iam service-accounts add-iam-policy-binding \
  "external-secrets-dev@${PROJECT_ID}.iam.gserviceaccount.com" \
  --project="${PROJECT_ID}" \
  --role="roles/iam.workloadIdentityUser" \
  --member="serviceAccount:${PROJECT_ID}.svc.id.goog[external-secrets/external-secrets]"
```

Repeat for UAT and prod:

```text
external-secrets-uat@PROJECT_ID.iam.gserviceaccount.com
external-secrets-prod@PROJECT_ID.iam.gserviceaccount.com
```

## 10. Create A ClusterSecretStore In Each Target Cluster

Create one `ClusterSecretStore` per target cluster.

Dev:

```yaml
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: gcp-secret-store-dev
spec:
  provider:
    gcpsm:
      projectID: montra-490412
      auth:
        workloadIdentity:
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
```

Apply it to `dev-cluster`:

```bash
kubectl apply -f cluster-secret-store-dev.yaml
```

UAT:

```yaml
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: gcp-secret-store-uat
spec:
  provider:
    gcpsm:
      projectID: montra-490412
      auth:
        workloadIdentity:
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
```

Prod:

```yaml
apiVersion: external-secrets.io/v1
kind: ClusterSecretStore
metadata:
  name: gcp-secret-store-prod
spec:
  provider:
    gcpsm:
      projectID: montra-490412
      auth:
        workloadIdentity:
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
```

Verify:

```bash
kubectl get clustersecretstore
kubectl describe clustersecretstore gcp-secret-store-dev
```

If you want stronger namespace isolation, use namespace-scoped `SecretStore` instead of `ClusterSecretStore`. For your 30+ app setup, `ClusterSecretStore` is simpler and easier to operate.

## 11. Add ExternalSecret Support To The Shared Helm Chart

Add defaults to `gitops-repo/helm/values.yaml`:

```yaml
externalSecrets:
  enabled: false
  refreshInterval: 1h
  secretStoreName: ""
  secretStoreKind: ClusterSecretStore
  targetName: ""
  creationPolicy: Owner
  data: []
  # Example:
  # data:
  #   - secretKey: JWT_SECRET
  #     remoteKey: spring-boot-realworld-jwt-secret
  #   - secretKey: DB_PASSWORD
  #     remoteKey: spring-boot-realworld-db-password
```

Create `gitops-repo/helm/templates/externalsecret.yaml`:

```yaml
{{- if .Values.externalSecrets.enabled }}
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: {{ required "externalSecrets.targetName is required" .Values.externalSecrets.targetName | quote }}
  namespace: {{ .Values.global.namespace | quote }}
  labels:
    {{- include "microservice.labels" . | nindent 4 }}
spec:
  refreshInterval: {{ .Values.externalSecrets.refreshInterval | default "1h" | quote }}
  secretStoreRef:
    name: {{ required "externalSecrets.secretStoreName is required" .Values.externalSecrets.secretStoreName | quote }}
    kind: {{ .Values.externalSecrets.secretStoreKind | default "ClusterSecretStore" | quote }}
  target:
    name: {{ required "externalSecrets.targetName is required" .Values.externalSecrets.targetName | quote }}
    creationPolicy: {{ .Values.externalSecrets.creationPolicy | default "Owner" | quote }}
  data:
    {{- range .Values.externalSecrets.data }}
    - secretKey: {{ required "externalSecrets.data[].secretKey is required" .secretKey | quote }}
      remoteRef:
        key: {{ required "externalSecrets.data[].remoteKey is required" .remoteKey | quote }}
        {{- if .version }}
        version: {{ .version | quote }}
        {{- end }}
        {{- if .property }}
        property: {{ .property | quote }}
        {{- end }}
    {{- end }}
{{- end }}
```

Then update `gitops-repo/helm/templates/deployment.yaml` so the container reads the generated Kubernetes Secret.

Add this under the container, beside the existing `env:` block:

```yaml
          {{- if .Values.externalSecrets.enabled }}
          envFrom:
            - secretRef:
                name: {{ required "externalSecrets.targetName is required" .Values.externalSecrets.targetName | quote }}
          {{- end }}
```

With `envFrom`, every key in the generated Kubernetes Secret becomes an environment variable in the pod.

Example:

```text
Secret key JWT_SECRET -> pod env var JWT_SECRET
Secret key DB_PASSWORD -> pod env var DB_PASSWORD
```

## 12. Configure One App Values File

Example: `gitops-repo/applications/dev/spring-boot-realworld-values.yaml`

```yaml
global:
  replicaCount: 1
  namespace: dev
  fullnameOverride: java-dev
  nameOverride: java-dev

deployment:
  image:
    name: "asia-south2-docker.pkg.dev/montra-490412/dev-app-images/spring-boot-realworld-app"
    tag: "72cd925"

externalSecrets:
  enabled: true
  secretStoreName: gcp-secret-store-dev
  secretStoreKind: ClusterSecretStore
  targetName: spring-boot-realworld-secrets
  refreshInterval: 1h
  data:
    - secretKey: JWT_SECRET
      remoteKey: spring-boot-realworld-jwt-secret
    - secretKey: DB_PASSWORD
      remoteKey: spring-boot-realworld-db-password
```

Important repo-specific note:

Your current chart reads `nameOverride` and `fullnameOverride` from `global.nameOverride` and `global.fullnameOverride`. Top-level `nameOverride` and `fullnameOverride` will not affect the rendered resource names unless the chart helpers are changed.

## 13. What ArgoCD Will Apply

When ArgoCD syncs an app, Helm renders:

```text
Deployment
Service
ExternalSecret
```

ESO then sees the `ExternalSecret` and creates:

```text
Kubernetes Secret
```

That generated Kubernetes Secret is not stored in Git. It is owned by ESO.

The pod receives secret values because the Deployment has:

```yaml
envFrom:
  - secretRef:
      name: spring-boot-realworld-secrets
```

## 14. ArgoCD Sync Order

Recommended order:

```text
1. External Secrets Operator
2. ClusterSecretStore
3. Application workloads
```

If managing these with ArgoCD, use sync waves:

```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "-10"
```

Suggested waves:

```text
ESO install app       -> -20
ClusterSecretStore   -> -10
Application workloads -> 0
```

## 15. Validate End To End

Check the `ExternalSecret`:

```bash
kubectl get externalsecret -n dev
kubectl describe externalsecret spring-boot-realworld-secrets -n dev
```

Expected condition:

```text
Type: Ready
Status: True
Reason: SecretSynced
```

Check the generated Kubernetes Secret exists:

```bash
kubectl get secret spring-boot-realworld-secrets -n dev
```

Check that a specific key exists without printing the secret value:

```bash
kubectl get secret spring-boot-realworld-secrets -n dev \
  -o jsonpath='{.data.JWT_SECRET}' | wc -c
```

A number greater than `0` means the key exists.

Check the pod got restarted after adding the secret reference:

```bash
kubectl rollout status deployment/java-dev-app -n dev
```

Use the actual rendered Deployment name if different.

## 16. Common Errors And Fixes

`ExternalSecret` says `SecretSyncedError`:

```text
Most likely remoteKey is wrong, the Google Secret Manager secret does not exist,
or ESO does not have permission to access it.
```

`permission denied` or `403` in ESO logs:

```text
Check roles/secretmanager.secretAccessor on the Google service account.
Check roles/iam.workloadIdentityUser binding for the Kubernetes service account.
Check the Helm serviceAccount annotation on ESO.
```

Secret exists but pod env var is missing:

```text
Restart the Deployment. Kubernetes does not update environment variables inside
already-running containers when a Secret changes.
```

Secret rotation happened in Google Secret Manager but app still uses old value:

```text
ESO updates the Kubernetes Secret, but pods need a restart to pick up env var
changes. Use a manual rollout restart or install a controller like stakater/reloader.
```

ApplicationSet cannot find values file:

```text
Your ApplicationSet currently expects ../applications/{{env}}/{{app}}-values.yaml.
Make sure every app values file uses .yaml, not .yml, or update the ApplicationSet.
```

## 18. If The Cluster Is Actually EKS

This guide assumes GKE because your clusters and images are in Google Cloud.

If the target cluster is really EKS, do not use the GKE Workload Identity steps. Use Google Workload Identity Federation for an external Kubernetes or AWS identity, or use a Google service account JSON key as a fallback. The JSON key approach works, but it is less secure because it creates a long-lived credential in Kubernetes.
